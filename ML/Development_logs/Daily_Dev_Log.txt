Summary:
Today covered the core Phase 3 ML implementation: dataset analysis, feature
engineering, linear regression training from scratch, simulator integration,
and the first version of the dashboard reality toggle.
 
Work completed:
 
1) EDA and data preparation (ML/)
- Analyzed the Sacramento DPV dataset (NSRDB Location ID 151015, 19 MW
  installation, 2006) using eda.py, generating 16 exploratory figures covering
  distributions, correlations, and temporal patterns.
- Identified and resolved a critical timezone mismatch between the Actual Power
  CSV (local time) and the Weather CSV (UTC), which had inverted all GHI
  correlations from -0.35 to +0.96 after correction.
- Built the full cleaning pipeline in feature_engineering.py: resampled Actual
  Power from 5-minute to 30-minute intervals, merged on datetime, engineered a
  Cloud_Cover_Ratio feature (GHI / ClearskyGHI), filtered nighttime rows
  (zenith >= 90 degrees), and saved 8,875 clean daytime rows to
  ML/output/sacramento_clean.csv.
 
2) Linear regression model from scratch (ML/train.py)
- Implemented gradient descent from scratch without scikit-learn or any ML
  library, as required by the assignment.
- Ran 29 feature combination experiments across four groups: single features,
  domain groupings, progressive addition, and leave-one-out analysis.
- Selected model G4_drop_zenith (7 features: GHI, DNI, DHI, temperature,
  relative humidity, cloud type, Cloud_Cover_Ratio) based on test RMSE of
  1.676 MW and R2 of 0.792.
- Compared against two baselines on a fair daytime-only evaluation window:
  daily average (MAE 3.049 MW) and 4-hour moving average (MAE 1.558 MW).
  Selected model achieved MAE of 1.354 MW, outperforming both.
- Saved model weights to ML/output/model_weights.json.
 
3) SolarPredictor bridge class (ML/solar_predictor.py)
- Built SolarPredictor as a bridge between the trained model and the simulator.
- Loads model_weights.json and sacramento_clean.csv at initialization.
- Builds a (month, hour) lookup table with 155 combinations covering daytime
  hours across all months.
- Returns a capacity factor per (hour_of_day, day_of_year) call, which the
  simulator multiplies by panel count and peak power to produce realistic
  hourly generation in kW.
 
4) Simulator integration
- Fixed path resolution in Simulation.py: the ML directory is three levels up
  from Simulator/src/, not two. Changed dirname call count accordingly.
- Fixed HouseholdSimulation._build_household_config(): the use_ml_solar flag
  was never passed to individual household configs, causing the ML predictor
  to silently fall back to math.sin. Added the missing key to the returned dict.
- Updated main.py to present a config selection submenu when neighborhood mode
  is chosen, allowing the user to select between standard and ML configs.
- Updated data_processor.py default paths to point to the two ML simulation
  results folders.
 
5) ML simulation runs
- Ran well_designed scenario with ML solar (sim_20260513_133044): 24
  households, 360 days, LOAD_PRIORITY. Self-sufficiency averaged 99.9%,
  net neighborhood revenue of $3,032.
- Ran unadvised scenario with ML solar (sim_20260513_164753): same setup
  with subdimensioned system sizing. Self-sufficiency averaged 68.4%,
  near break-even at -$26 net.
- Generated data_ml/ folder with the five dashboard JSONs using
  data_processor.py.
 
6) Dashboard reality toggle (first version)
- Added mlMode: false to the global state object in core.js.
- Refactored data.js to use a central loadData(folder) function, replacing
  the original hardcoded fetch block.
- Added wireRealityToggle() and updateRealityToggleUI() functions to data.js.
- Added the reality toggle section to index.html: dark background panel at
  the bottom of the page with narrative text and a mode-switching button.
 
Notes / Issues:
- The 2006 Sacramento training data is used to simulate a 2024 scenario. This
  is acceptable because the SolarPredictor aggregates by (month, hour),
  removing year-specific anomalies and representing typical seasonal behavior.
- Sacramento's high solar resource caused the unadvised ML scenario to perform
  significantly better than the synthetic unadvised model, shifting the
  narrative from loss vs. revenue to break-even vs. significant revenue.
